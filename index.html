    Welcome to Cortez

    What is Cortez?

    Cortez is a design suite for interactive adventures; it’s a web/mobile app that allows you to create/view location-based information for exploration.

    More specifically, Cortez is the Senior Design project for EECS 582 under construction by Joe Champion, Samuel Lamb, Sam Groth, Roxanne Calderon, Gehrig Keane, and Austin Sievert.

    Cortez has both a web and mobile component to help easily create and view custom maps, using a Google Maps interface to allow map creators to place pins with information and media, which can then be viewed by mobile users physically at that location.

    What does Cortez do?

    Cortez has come a long way since its conception last semester. We now have fully fleshed mobile and web applications which are able to communicate with each other.

    The web application first implemented a map and pin system. The map uses Google map so the user can explore different locations across the globe. The on-click pin system allows map creators to mark points of interest on the map with a description and add accompanying media (pictured, audio, video, or text). Combined, these two systems provide an interface for curators to build custom maps suited to their needs and interests. The web interface also creates JSON interfaces to export to the mobile console.

    On the mobile app, we have designed a system for displaying the maps created from the web frontend. A Google map displays the pins, which can be selected to display their associated information. This map ties into the device’s GPS functionality, allowing the user to navigate the map in real time, walking from pin to pin. The pins will pop in and out as the user views relevant portions of the map. The pins themselves are pulled from a local JSON file. This means that the user will be able to choose which map to use (which will tie into the map and pin system being developed on the web application).

    What problems did we have?

    The latest hurdle to web app creation was the implementation of S3 into the Heroku component. Heroku would not save the long term media elements (images, video, audio) that we required with our current Rails gem, Paperclip. As such, we had to implement an AWS S3 component to store these items.

    The mobile app hasn’t been without its share of difficulties, either. Android documentation is sometimes confusing or out of date, requiring our developers to be especially wary of using deprecated function calls. Throughout development, we have continued to experience temperamental behavior of Android Studio (the official IDE) across differing platforms.

    The app has called for an ever-increasing list of concepts and Java boilerplate:

    • Android application lifecycle (onCreate, onStart, onPause, onStop), and order of execution that resembles programming for embedded systems
    • Network handling, and asynchronous downloading tasks in RESTful fashion
    • Context switching using Android Intent and BroadcastReceiver, and data / parameter passing across those contexts for a dynamic user experience
    • Android and Google Play Services to monitor user location and serve map media

    Despite the formidable challenges mentioned above, our mobile component is well on its way to being feature-complete!

    Next Steps

    We have a lot ahead of us regarding the development of Cortez. Both the web and mobile applications have come a long way, but there is so much more we can do.

    Our next web goal is to fully integrate the users component to allow private and public maps. We also hope to implement a scavenger hunt mode in the next few weeks. At that point, the web component will have completed its feasible goals for the semester.

    Current mobile goals include synchronizing lists between the app and the database, streamlining our download process, and supporting more media types as they are added to the database. Our stretch goals include a user credentials system for private maps, and possibly map creation or modification from the mobile app.
